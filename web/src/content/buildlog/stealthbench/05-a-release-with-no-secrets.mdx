Last update I cut packaging, and I said why: `pip install stealthbench` would hand you a command that can't actually run. This update I shipped the release machinery anyway. Not because the objection stopped being true. Because I found a shape for the package that doesn't lie about it, and the part that turned out to be worth writing up is a GitHub Actions release that publishes with no secret stored anywhere in the repo.

**stealthbench** scores browser-automation stealth setups (configs) against open, self-hosted bot detectors on my own `localhost`, and commits every result. It's a benchmark you *run*, not a library you import. That's exactly what made packaging awkward, and it's the tension I walked out on in [update 4](/building/stealthbench/04-one-command-to-run-it/): a wheel on an index implies "install this and go," and this thing can't go on its own.

## Why package a tool that can't stand alone

The wheel is the runner and the CLI, nothing more. A real run still needs a browser stack (Chrome or Chromium, Camoufox's patched Firefox, Node.js) and the two detector servers up on `localhost`. None of that fits in a Python package, and pretending otherwise is the small dishonesty I refused in update 4.

So the resolution isn't clever, it's blunt: the most honest artifact this sprint produced is a five-line warning in the README that tells you the package can't do the thing its name implies.

> `pip install` ships the runner + CLI only. `pipx install --index-url https://test.pypi.org/simple/ stealthbench` (TestPyPI) gives you the `stealthbench` and `stealthbench-serve` commands, but a real run still needs a browser stack and the detector servers. Install ≠ ready-to-run.

That last line is the same claim from update 4, moved out of my prose and into the shipped docs. And note where it points: **TestPyPI**, not real PyPI. I wired the whole release pipeline and aimed it at `test.pypi.org`, then stopped. The machinery is real and proven, without me planting a "just pip install me" entry on the index everyone actually searches. A package that warns you it isn't ready has no business sitting on the front index yet.

## A release with no token in it

Here's the piece I didn't expect to enjoy. The usual way to publish from CI is to generate a PyPI API token, paste it into your repository secrets, and hope it never leaks and that you remember to rotate it. I didn't want a long-lived credential living in the repo's settings for a project whose entire pitch is "nothing here touches a service it shouldn't."

The pattern that removes it is **OpenID Connect trusted publishing**. Instead of a stored token, GitHub mints a short-lived identity token for the specific workflow run, and PyPI (or TestPyPI) verifies that it came from this repository, this workflow, and this environment before handing back a one-time credential to upload with. Nothing long-lived is stored on either side. The whole trust relationship is configured once on the index and asserted per run.

In the workflow that collapses to three load-bearing lines:

```yaml
on:
  push:
    tags: ['v*']            # a tag is the only trigger

jobs:
  build-and-publish:
    environment: testpypi
    permissions:
      id-token: write        # let the job mint an OIDC identity, no stored token
    steps:
      - run: uv build        # clean sdist + wheel
      - uses: pypa/gh-action-pypi-publish@release/v1
        with:
          repository-url: https://test.pypi.org/legacy/
```

`id-token: write` is the whole trick: it grants the job permission to request the identity token, and the `pypa/gh-action-pypi-publish` action does the exchange for me. There's no `password:` field, no `secrets.PYPI_TOKEN`, nothing to leak in a log or forget in a rotation I'd inevitably skip. A secret you never store is one you can't spill.

## Two commands get real names

The smaller change: `pyproject.toml` now declares console scripts, so the `python -m` invocations from the last two updates become plain commands.

```toml
[project.scripts]
stealthbench = "stealthbench.__main__:main"
stealthbench-serve = "stealthbench.serve:main"
```

`stealthbench` is the bench runner, `stealthbench-serve` is the one-command detector bring-up I built in update 4. `uv build` turns them into a clean sdist and wheel, and `dist/` is git-ignored, unlike `results/`, which stays committed for traceability. Build output isn't a fact about the world; a benchmark run is.

## The button I wired and didn't press

Here's the honest state of it. `git tag` returns nothing. The release triggers only on a `v*` tag, so the workflow has never run once. The only build that exists anywhere is the one `uv build` produced on my laptop, sitting in that ignored `dist/`. Nothing has been published, not even to TestPyPI.

So the sprint landed console scripts, a wheel that builds clean, a release pipeline that needs no secret, and a README that warns you off, and then it stopped short of shipping. That reads like unfinished work, and in a way it is. But it's also the consistent move: a package that isn't ready to run has no business getting tagged and pushed to an index just to prove the pipeline turns over. I built the release and left it un-fired on purpose. The tag is the one manual step I want to stay manual until install actually equals ready-to-run.

That gate is the same thread I've carried since update 2: a headful CI run so the snapshot regenerates on a schedule instead of when I remember to. The release workflow now sits right next to the CI it'll eventually share a runner with. When a clean-room CI job can bring the whole stack up and produce a real number, the package will have earned the tag.

If you publish Python packages: have you moved to trusted publishing yet, or is there still an API token sitting in your repo secrets? I'm sold on the no-token release, and genuinely unsure where the line is between "wire it against TestPyPI and stop" and "just cut the real release." The repo is at [github.com/bessavagner/stealthbench](https://github.com/bessavagner/stealthbench).
