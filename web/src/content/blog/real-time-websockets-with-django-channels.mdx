Most of what we build on the web is request/response: the browser asks, the server answers, the connection closes. It's a beautiful model for fetching a page or saving a form, and it's the model Django was built around. But it has one stubborn limitation — the server can never speak first. If something happens on the server that a user should see *right now* — a chat message from someone else, a job finishing, a price ticking — plain HTTP has no way to tell the browser. The browser has to ask.

For a long time the workaround was to keep asking: poll the server every few seconds and hope the news is fresh. It works, but it's wasteful and it's slow, and it never feels truly live. This post is about the alternative — a real, persistent, two-way connection — and how [Django Channels](https://channels.readthedocs.io/) bolts it onto an otherwise ordinary Django app.

## Why request/response can't push

The reason Django can't push isn't a missing feature; it's the shape of the interface underneath it. Classic Django runs under [WSGI](https://peps.python.org/pep-3333/), and a WSGI application is, as the ASGI spec puts it, "a single, synchronous callable that takes a request and returns a response." One request in, one response out, done. There is no place in that contract for the server to hold a line open and send a second, unprompted message later. As the [ASGI introduction](https://asgi.readthedocs.io/en/latest/introduction.html) says plainly, that "single-callable interface just isn't suitable for more involved Web protocols like WebSocket."

So if you want push, you poll. The cost of polling is latency you can't avoid: an event that happens just after a poll has to wait for the *next* poll before anyone sees it. Crank the interval down and you flood the server with mostly-empty requests; crank it up and the app feels sluggish. There's no setting that's actually good.

[ASGI](https://asgi.readthedocs.io/en/latest/) — the Asynchronous Server Gateway Interface — is the way out. It's the "spiritual successor to WSGI," and it reshapes the contract: instead of one synchronous callable, an application is an async callable handed a `scope` (what kind of connection is this), a `receive` (pull the next incoming event), and a `send` (push an event out). That structure allows "multiple incoming and outgoing events" and "long-lived connections" — exactly what a WebSocket needs. Django supports being [deployed under ASGI](https://docs.djangoproject.com/en/5.1/howto/deployment/asgi/), and Channels is the library that uses that capability to add WebSockets to your project.

## Consumers and the channel layer

Channels gives you two main pieces. The first is the **consumer** — the WebSocket equivalent of a view. Instead of writing a raw ASGI event loop, you subclass `AsyncWebsocketConsumer` and fill in three lifecycle methods, which the [consumers docs](https://channels.readthedocs.io/en/latest/topics/consumers.html) describe as: `connect` ("called on connection"), `receive` ("called with either text_data or bytes_data for each frame"), and `disconnect` ("called when the socket closes"). The framework structures your code "as a series of functions to be called whenever an event happens, rather than making you write an event loop."

The second piece is the **channel layer** — a shared message bus, usually backed by Redis, that lets consumers (in different processes, on different machines) talk to each other. This is the part that makes broadcast possible. One browser's message arrives at one consumer instance; for every *other* browser to see it, that consumer has to fan it out through the layer.

Here's the `connect` method from a chatbot I built. When a socket opens, it accepts the connection and joins a named group, so it's now a subscriber to anything broadcast there:

```python
class AsyncChatConsumer(AsyncWebsocketConsumer):

    group_name = CHAT_GROUP_NAME

    async def connect(self):
        await self.accept()
        await self.channel_layer.group_add(
            self.group_name, self.channel_name
        )
        # ... load history and greet the client ...

    async def disconnect(self, close_code):
        await self.channel_layer.group_discard(
            self.group_name, self.channel_name
        )
```

`self.channel_name` is this specific connection's unique address; `group_add` adds it to the group. The symmetry matters: whatever you add in `connect`, you remove in `disconnect` with `group_discard`, or you leak dead connections into the group and keep trying to send to sockets that closed long ago.

## Broadcasting to a group

The payoff is in `receive`. When a client sends a message, the consumer doesn't reply directly — it hands the message to the channel layer with `group_send`, and the layer delivers it to *every* member of the group:

```python
async def receive(self, text_data):
    data = json.loads(text_data)
    response = "FAILED"
    model = ''
    if "chat" in data:
        message = data["chat"]["message"]
        model = data["chat"]["model"]
        response = await self.chat.ask(message, model=model)
    await self.channel_layer.group_send(
        self.group_name, {
            "type": "notify.client",
            "data": {
                "response": {"message": str(response)},
                "model": model if model else 'Bot',
            },
        }
    )

async def notify_client(self, event):
    await self.send(json.dumps({"chat": event["data"]}))
```

The clever bit is `"type": "notify.client"`. The channel layer takes that type, swaps the dot for an underscore, and calls a method of that name — `notify_client` — on *every* consumer in the group. Each of those consumers then does its own `self.send(...)` over its own socket. So one inbound message produces an outbound message to everyone connected. That's the broadcast, and it's why you can't just `await self.send()` from inside `receive`: that would only answer the one client who spoke. The round-trip through the layer is what reaches the rest.

On the browser side there's nothing exotic — the native [WebSocket API](https://developer.mozilla.org/en-US/docs/Web/API/WebSocket) is a few lines:

```javascript
const socket = new WebSocket(`wss://${location.host}/ws/chat/${uuid}/`);

socket.addEventListener("open", () => {
  socket.send(JSON.stringify({ chat: { message: "hello", model: "gpt" } }));
});

socket.addEventListener("message", (event) => {
  const { chat } = JSON.parse(event.data);
  renderMessage(chat); // the server pushed this — no request was made
});
```

That `message` handler fires whenever the server pushes, with no request on the client's part. That's the whole difference from polling, in one event listener.

## The sync/async ORM boundary

There's a trap waiting inside every async consumer. Your consumer methods are `async`, but Django's ORM is not. The [Channels database docs](https://channels.readthedocs.io/en/latest/topics/databases.html) are blunt about it: "The Django ORM is a synchronous piece of code," and calling it directly from async code can corrupt connection state. Django's own [ASGI guide](https://docs.djangoproject.com/en/5.1/howto/deployment/asgi/) repeats the rule: "Do not call blocking synchronous functions or libraries in any async code."

The fix is `database_sync_to_async`, which the docs describe as "a version of `asgiref.sync.sync_to_async` that also cleans up database connections" when it's done. You wrap any ORM access in it — as a decorator on a helper method, or inline — and it runs the query in a thread, hands the result back to your coroutine, and tidies up the connection afterward:

```python
from channels.db import database_sync_to_async

@database_sync_to_async
def get_chat(self):
    return ContextChat.objects.get(
        uuid=self.scope["url_route"]["kwargs"]["uuid"]
    )
```

Now `await self.get_chat()` is safe inside `connect` or `receive`. The mental model that keeps you out of trouble: the consumer lives in the async world, the ORM lives in the sync world, and `database_sync_to_async` is the only sanctioned door between them. Reach through it every time you touch the database, and never just *call* a synchronous query from a coroutine and hope.

## Measuring push vs. poll

So how much does pushing actually buy you? The honest answer needs a real network and a real load test, but the *shape* of the win is easy to reason about, and worth picturing. Below is a synthetic timing model — not a benchmark of any server — comparing when a client learns about an event under each transport. WebSocket push pays one network hop. HTTP polling pays a network hop *plus* however long the event sat waiting for the next poll, which is uniform between zero and the full polling interval.

<Image src={latency} alt="Synthetic delivery latency: WebSocket push clusters tightly near a single network hop, while 3-second HTTP polling spreads events out to roughly a second and a half on average — push is both faster and far more consistent." />

The push distribution is a tight spike near the network hop; the polling distribution is smeared flat across the whole interval, with a median around half of it. The lesson isn't the exact milliseconds — it's that polling's latency is dominated by *waiting for permission to ask*, and that's precisely the cost a persistent connection deletes.

A persistent connection isn't free, though. The browser [`WebSocket` API has no backpressure](https://developer.mozilla.org/en-US/docs/Web/API/WebSocket): if the server pushes faster than the client drains, messages pile up in a buffer and memory grows. The server side has its own version of this — Channels gives each channel a [capacity](https://channels.readthedocs.io/en/stable/channel_layer_spec.html), and "when a channel is at or over capacity, trying to `send()` to that channel may raise `ChannelFull`." Group sends, notably, never raise — they "silently drop the message if over capacity," following ASGI's at-most-once delivery policy. So broadcasting is lossy under pressure by design, which is fine for a chat ping and very much not fine for, say, a financial ledger. Knowing which one you're building is the whole job.

## Where I used this

I built exactly this on a Django chatbot project: an `AsyncWebsocketConsumer`, a single Redis-backed group, and `database_sync_to_async` around every ORM call so loading chat history and saving messages never blocked the event loop. The result is a chat that feels instant and stays in sync across tabs and users — no polling, no "refresh to see new messages," just the server speaking the moment it has something to say.

The pieces generalize well beyond chat. Any time the server knows something before the client could think to ask — a notification, a live dashboard tile, a long job finishing — the same three moves apply: a consumer that joins a group on connect, a `group_send` to fan an event out, and a disciplined sync/async boundary around the database. Get those right and "real-time" stops being a feature you bolt on and becomes just how the app talks.
